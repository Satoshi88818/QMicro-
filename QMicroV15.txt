qmicro/src/hw/profiling.rs

v15 Changes:

Critical fix: Producer::push now writes data to the buffer slot before publishing the incremented write_ptr. Under the previous v14 design (fetch_add first, then write_volatile), an interrupt firing between those two instructions could cause a Consumer to observe a non-empty queue and read uninitialised/stale memory.

All SeqCst replaced with Release on writes and Acquire on reads — the mathematically sufficient ordering for an SPSC queue, and critically cheaper on ARM (avoids DMB ISH in profiling hot paths).

mark_start/mark_end profiling atomics also downgraded from SeqCst to Relaxed/Release/Acquire where appropriate.

Extended Kani verification with verify_ring_buffer_math proof for power-of-two modulo safety.

qmicro/src/hw/telemetry.rs

v15 Changes:

RECORD_BYTES padded from 9 → 16 bytes (alignment-friendly, power-of-two-safe).

STAGING_BYTES is now 512 (32 × 16), a power of two. When UART_WR wraps at u32::MAX, % 512 wraps cleanly with no discontinuity.

cortex_m::asm::dmb() inserted after payload copy, before program_uart_dma — ensures the CPU write buffer is flushed to SRAM before the DMA bus master reads from it.

All SeqCst in the hot drain/flush path replaced with Acquire/Release.

Architecture Notes

Why write-before-publish matters on Cortex-M

On ARM Cortex-M, ISRs can preempt between any two instructions. The v14 sequence was:

1. fetch_add(write_ptr) ← ISR fires here 2. write_volatile(slot) 

Any ISR-driven consumer that fires at step 1 would see write_ptr > read_ptr and call read_volatile on a slot that hasn't been written yet — returning stale initialiser data. The v15 sequence is:

1. load(write_ptr) ← snapshot only, no visible side-effect 2. write_volatile(slot) ← data committed 3. store(write_ptr + 1, Release) ← pointer published with Release 

No observer can detect the queue is non-empty until after the data is committed.

Why power-of-two STAGING_BYTES matters

AtomicU32 wraps at u32::MAX. If STAGING_BYTES is not a power of two:

base = (u32::MAX % 288) = 255 ← last valid position next = (0 % 288) = 0 ← next position after wrap 

The apparent "next slot" jumps backward by 255 bytes, causing the new record to overwrite an arbitrary range of the staging buffer — a silent DMA stream corruption. With STAGING_BYTES = 512, x % 512 == x & 511 for all x, and the wrap from u32::MAX to 0 is perfectly continuous.
